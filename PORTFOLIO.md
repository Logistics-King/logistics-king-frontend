# 택배왕 프론트엔드 포트폴리오 정리

## 프로젝트 개요

택배왕은 화주와 택배 대리점이 온라인에서 계약 요청, 단가 제안, 단가 조율, 최종 계약까지 진행할 수 있는 물류 계약 플랫폼입니다.

프론트엔드 작업의 핵심은 단순 화면 구현이 아니라, 자주 변경되는 백엔드 API와 복잡한 물류 계약 도메인을 사용자 흐름에 맞게 안정적으로 연결하는 것이었습니다.

## 핵심 기여

- HttpOnly Cookie 기반 인증 흐름과 공통 API 클라이언트 구현
- 화주/대리점/기사/관리자 역할별 라우팅과 권한별 진입 흐름 설계
- 계약 요청의 단일 품목 구조를 다중 배송 품목 라인 구조로 전환
- 대리점 제안과 단가 조율을 배송 품목 라인별로 비교할 수 있게 개선
- 백엔드 API 스펙 변경에 대응 가능한 타입 기반 연동 구조 정리
- 프로필 미등록, 세션 만료, 권한 오류 같은 실제 서비스 예외 흐름 UX 개선

## 기술 스택

- Next.js 16 App Router
- React 19
- TypeScript
- Tailwind CSS
- REST API
- Server-Sent Events
- HttpOnly Cookie 인증
- Kakao 우편번호 서비스

## 엔지니어링 사례

### 1. HttpOnly Cookie 인증과 공통 API 클라이언트

백엔드는 access token과 refresh token을 HttpOnly Cookie로 내려주는 구조였습니다. 프론트에서 토큰을 직접 읽거나 localStorage에 저장하지 않는 방향으로 인증 흐름을 맞췄습니다.

해결한 문제:

- 보호 API마다 `credentials: "include"`가 빠지면 인증이 실패함
- 세션 만료 시 각 화면에서 401 처리 코드가 중복될 수 있음
- refresh token도 HttpOnly라 JS에서 직접 다루면 안 됨

구현:

- 모든 보호 API 호출을 공통 `apiFetch`로 래핑
- 401 응답 시 refresh API를 한 번 호출한 뒤 원 요청 재시도
- refresh 실패 시 홈으로 리다이렉트
- public API는 `credentials: "omit"`로 분리
- 백엔드 공통 응답 shape인 `payload.code`, `payload.errorMessage`, `payload.response`를 한 곳에서 처리

성과:

- 인증 처리 코드가 각 화면에 흩어지지 않음
- 토큰 저장 위치에 대한 보안 리스크를 줄임
- 백엔드 error code 기반 사용자 안내가 쉬워짐

## 2. API 스펙 변경에 강한 타입 구조

프로젝트 진행 중 백엔드 API 스펙이 여러 번 바뀌었습니다.

대표 변경:

- `boxSize`: 자유 문자열에서 enum으로 변경
- `coldChainType`: boolean 조건에서 `NONE | REFRIGERATED | FROZEN` enum으로 변경
- 계약 요청: 단일 대표 품목에서 `items[]` 다중 배송 품목 라인으로 변경
- 제안: 대표 단가에서 배송 품목 라인별 단가로 확장
- 최종 계약: 계약 당시 배송 품목 라인 snapshot 추가
- 계약 요청 일정: 단건/정기 계약 조건 추가

대응 방식:

- 백엔드 enum을 프론트 타입으로 고정
- feature별 API 파일에 request/response 타입을 명시
- nullable 응답과 과거 데이터를 방어적으로 처리
- `itemId`, `contractRequestItemId`처럼 전환기 필드는 fallback helper로 흡수
- 화면 컴포넌트가 원본 API shape에 과도하게 의존하지 않도록 표시용 formatter를 분리

예시:

기존 제안 데이터에는 `proposal.items[]`가 없을 수 있었습니다. 이 경우 계약 요청 상세의 `items[]`를 기준으로 배송 품목 라인을 만들고, 대표 단가를 각 라인의 fallback 단가로 표시했습니다.

성과:

- API 변경 후에도 화면 전체를 다시 만들지 않고 영향 범위를 좁혀 대응
- 신규 데이터와 과거 데이터가 섞여도 UI가 깨지지 않음
- 백엔드 스펙 handoff를 타입 변경으로 빠르게 반영 가능

## 3. 다중 배송 품목 라인 기반 계약 요청 구조

초기 계약 요청은 대표 품목 하나를 기준으로 처리됐지만, 실제 물류 계약은 하나의 계약 안에 여러 배송 품목이 포함될 수 있습니다.

문제:

- 같은 계약 안에서도 품목별 박스 규격, 수량, 온도 조건이 다를 수 있음
- 대표 품목 하나만 표시하면 대리점이 실제 물류 조건을 판단하기 어려움
- 품목마다 단가가 달라질 수 있는데 대표 단가만 있으면 비교가 부정확함

구현:

- 계약 요청 등록 폼을 `items[]` 중심으로 재구성
- `+ 품목 라인 추가` 시 직전 라인 값을 복사해 반복 입력 비용 감소
- 등록된 배송 품목 템플릿을 선택하면 품목명, 카테고리, 박스 규격, 수량, 온도 조건 자동 반영
- 이미 계약 요청에 묶인 배송 품목은 재선택 불가능하게 제외
- 총 박스 수와 총 낱개 수를 라인 합산으로 자동 계산
- top-level 대표 필드는 백엔드 호환용으로 첫 번째 item 기준 세팅

성과:

- 한 계약 안의 여러 배송 품목을 실제 업무 단위에 맞게 표현
- 대리점이 일감 조회에서 품목별 조건을 보고 제안 가능
- 추후 최종 계약과 기사 배정에서도 계약 당시 품목 snapshot을 기준으로 확장 가능

## 4. 배송 품목 라인별 제안 단가와 가격 조율

가장 큰 프론트 도메인 개선은 제안 단가를 대표 단가가 아니라 배송 품목 라인별로 볼 수 있게 만든 것입니다.

문제:

- 하나의 계약 요청에 여러 배송 품목이 있음
- 대리점 제안 화면에서 단가가 어떤 품목에 대한 것인지 불명확했음
- 사용자가 보기에는 여러 계약처럼 보이거나, 반대로 어떤 라인의 단가인지 알 수 없었음
- 가격 조율 이벤트도 대표 단가만 있으면 품목별 협상이 어려움

해결:

- `proposal.items[]`와 `contractRequest.items[]`를 `contractRequestItemId`로 매칭
- 대리점 내 제안 화면에서 일감 조회와 동일하게 품목명, 박스 규격, 수량, 온도 조건, 희망 단가 표시
- 품목별 제안 단가 입력 시 대표 단가를 라인 단가 기반으로 자동 계산
- 가격 조율 모달에서도 배송 품목 라인별 조율 단가 입력
- 협상 이벤트 조회 시 event item 단가를 라인별로 표시
- 과거 제안처럼 item 단가가 없는 경우 대표 단가 fallback 적용

성과:

- 대리점과 화주 모두 “어떤 배송 품목의 단가인지” 명확히 판단 가능
- 대표 단가 중심 UI에서 품목 라인 중심 UI로 전환
- 가격 협상 UX가 실제 계약 단가 산정 방식에 가까워짐

## 5. 단건/정기 계약 일정 조건 반영

백엔드에 계약 요청 일정 조건이 추가되면서 프론트도 단건 계약과 정기 계약을 구분해야 했습니다.

요구사항:

- 단건 계약: 회수 희망 기간, 배송 희망 기간
- 정기 계약: 주간/월간 회수 규칙
- 주간: 요일 다중 선택
- 월간: 매월 회수일 입력
- 기존 요청은 기본적으로 `SINGLE`로 동작

구현:

- 계약 요청 등록/수정 폼에 `단건 / 정기` segmented control 추가
- `SINGLE` 선택 시 날짜 range 입력 노출
- `RECURRING` 선택 시 주기 selector 노출
- `WEEKLY` 선택 시 요일 multi-select 노출
- `MONTHLY` 선택 시 1~31일 입력 노출
- 날짜 범위와 정기 회수 조건 프론트 검증
- 화주 계약 요청 조회, 대리점 일감 조회, 최종 계약 목록에 일정 조건 표시

성과:

- 신규 API 필드가 기존 화면을 깨지 않도록 기본값과 nullable 처리
- 계약 요청 생성 시 일정 조건을 도메인 정책에 맞게 입력 가능
- 최종 계약 화면에서도 계약 요청 당시 일정 snapshot 확인 가능

## 6. 역할별 온보딩과 프로필 선등록 흐름

화주와 대리점은 계정만으로 주요 기능을 사용할 수 없습니다. 화주는 사업자/주소 정보가 있어야 배송 품목과 계약 요청을 만들 수 있고, 대리점은 영업 거점과 담당 지역이 있어야 일감 조회와 제안을 할 수 있습니다.

문제:

- 프로필 없이 기능 화면에 접근하면 백엔드에서 `VENDOR_NOT_FOUND`, `AGENCY_NOT_FOUND` 발생
- 단순 빨간 오류 문구만으로는 사용자가 다음 행동을 알기 어려움
- 로그인 후 역할별로 가야 하는 화면이 다름

구현:

- 로그인 성공 후 사용자 role 확인
- role별 프로필 조회 API 호출
- 프로필이 없으면 해당 프로필 등록 페이지로 이동
- 기능 화면에서도 프로필 미등록 상태를 CTA가 있는 안내 카드로 표시
- `AGENCY_ALREADY_EXISTS`, `AGENCY_NOT_FOUND` 같은 상태별 흐름 분기

성과:

- 기능 접근 전 필수 데이터 등록을 자연스럽게 유도
- 사용자 입장에서 “왜 안 되는지”보다 “무엇을 해야 하는지”가 명확해짐
- 권한과 프로필 상태가 섞인 오류를 프론트 흐름으로 정리

## 7. 알림 링크 라우팅과 SSE 실시간 수신

알림 기능에서는 단순 목록 표시보다 라우팅과 상태 동기화가 중요했습니다.

문제:

- 백엔드 `linkUrl`은 리소스 중심 경로라 프론트 라우트와 맞지 않아 404가 발생할 수 있음
- 알림 클릭 시 사용자의 role에 따라 이동해야 하는 화면이 다름
- SSE 이벤트는 REST 공통 응답 wrapper 없이 내려옴

구현:

- 알림 클릭 시 `linkUrl`, `referenceType`, 현재 사용자 role을 함께 판단
- 백엔드 경로를 프론트 경로로 매핑
- 계약 요청 알림은 query parameter로 상세 패널 자동 오픈
- `EventSource`에 `withCredentials: true` 적용
- SSE 수신 시 unread badge 갱신 또는 목록 재조회
- 재연결/누락 가능성을 고려해 unread count 재조회 루틴 유지

성과:

- 알림 클릭 404 문제 해결
- 새 제안/계약 이벤트를 실시간으로 사용자에게 전달
- REST 목록 API를 원천 상태로 두고 SSE는 전달 채널로 분리

## 8. 입력 품질과 운영형 폼 UX 개선

물류 계약 서비스는 폼 입력이 많고, 잘못된 입력이 API 오류나 운영 데이터 오류로 이어질 수 있습니다.

개선한 입력:

- 사업자등록번호: `123-45-67890` 자동 포맷, 10자리 제한
- 연락처: `010-1234-5678` 자동 포맷, 과도한 자리수 입력 차단
- 금액: 천 단위 콤마 표시
- 무게: 천 단위 콤마 표시
- 주소: Kakao 우편번호 검색 연동
- 비밀번호 확인: 불일치 시 제출 방지
- 박스 규격/온도 조건: free text 대신 enum select

성과:

- 백엔드 validation 이전에 명확한 입력 제약 제공
- 사용자가 입력 형식을 외우지 않아도 되는 폼 UX 구성
- 운영 데이터의 형식 일관성 확보

## 트러블슈팅 요약

### 문제 1. 대리점 내 제안에서 단가가 어떤 품목 기준인지 알 수 없음

원인:

- 제안 응답은 단가 item만 포함하고, 품목명/박스/수량 정보는 계약 요청에 있음

해결:

- 제안 목록 로드 후 `contractRequestId`로 계약 요청 상세 추가 조회
- `contractRequestItemId`로 proposal item과 contract request item 매칭
- proposal item이 없는 과거 데이터는 대표 단가 fallback 표시

### 문제 2. 알림 클릭 시 404 발생

원인:

- 백엔드 `linkUrl`과 프론트 라우트 체계가 다름

해결:

- role, referenceType, referenceId를 기준으로 프론트 라우트 변환
- 계약 요청 상세는 query parameter로 패널 자동 오픈

### 문제 3. API 스펙이 자주 바뀌며 화면이 쉽게 깨질 위험

원인:

- enum, nullable 필드, response object가 계속 변경됨

해결:

- feature API 타입을 먼저 수정하고 UI는 formatter/helper를 통해 표시
- 신규 필드는 optional/default 처리
- 이전 데이터와 신규 데이터를 동시에 처리하는 fallback 추가

## 이력서용 요약

Next.js와 TypeScript로 물류 계약 플랫폼 `택배왕` 프론트엔드를 구현했습니다. HttpOnly Cookie 인증 흐름, 권한별 라우팅, 프로필 온보딩, 다중 배송 품목 기반 계약 요청, 품목별 제안 단가, 가격 조율, 실시간 알림을 백엔드 API와 연동했습니다. 특히 계약 요청이 단일 대표 품목에서 다중 배송 품목 라인 구조로 확장되는 과정에서 타입 기반 API 대응, 과거 데이터 fallback, 품목별 단가 매칭 로직을 설계해 도메인 변경에 안정적으로 대응했습니다.

## 면접에서 강조할 포인트

### Q. 단순 CRUD와 다른 점은?

계약 요청 하나가 여러 배송 품목 라인을 포함하고, 각 라인마다 박스 규격, 수량, 온도 조건, 단가가 다를 수 있었습니다. 그래서 대표 품목 중심 UI가 아니라 배송 품목 라인 중심 UI로 구조를 바꿨고, 제안과 가격 조율도 `contractRequestItemId` 기준으로 매칭했습니다.

### Q. 가장 어려웠던 점은?

백엔드 API가 계속 변경되는 상황에서 기존 데이터를 깨지 않게 유지하는 점이었습니다. 신규 응답에는 `items[]`가 있지만 과거 데이터에는 없을 수 있었기 때문에, 계약 요청 상세를 추가 조회하고 대표 단가를 fallback으로 사용하는 방식으로 화면 안정성을 확보했습니다.

### Q. 보안이나 인증에서 고려한 점은?

토큰을 JS에서 읽지 않는 HttpOnly Cookie 구조를 유지했습니다. 모든 보호 API에 `credentials: "include"`를 적용하고, 401 발생 시 refresh 후 원 요청을 재시도하는 공통 API 클라이언트를 구현했습니다.

### Q. 사용자 경험에서 개선한 점은?

프로필이 없을 때 단순 오류가 아니라 등록 CTA를 보여주고, 사업자번호/연락처/금액/무게/주소처럼 운영 데이터 품질에 영향을 주는 입력은 자동 포맷과 제한을 적용했습니다. 또한 알림 클릭 시 백엔드 리소스 URL을 프론트 라우트로 변환해 404 없이 해당 업무 화면으로 이동하도록 했습니다.
