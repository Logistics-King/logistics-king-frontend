# 택배왕 프론트엔드 포트폴리오 정리

## 프로젝트 한 줄 소개

택배왕은 화주와 택배 대리점을 연결해 배송 계약 요청, 단가 제안, 계약 비교 과정을 온라인으로 관리하는 물류 계약 플랫폼입니다. 전화, 소개, 직접 영업 중심이던 택배 계약 과정을 웹 서비스로 전환하는 것을 목표로 했습니다.

## 문제 정의

기존 택배 계약은 화주가 여러 대리점에 직접 연락해 단가, 토요일 배송, 반품, 냉장/냉동 가능 여부를 반복 확인해야 했습니다. 대리점도 신규 화주를 확보하려면 직접 영업에 의존해야 했고, 화주의 물량과 상품 조건을 사전에 알기 어려웠습니다.

이 프로젝트는 다음 문제를 해결하려고 했습니다.

- 화주가 여러 대리점 조건을 한 화면에서 비교하기 어렵다.
- 대리점이 신규 화주 일감을 발견하고 제안할 채널이 부족하다.
- 계약 요청, 제안, 계약 진행 상태가 흩어져 있어 관리 비용이 크다.
- 권한별로 필요한 화면과 API가 달라 프론트에서 세션, 메뉴, 접근 흐름을 명확히 관리해야 한다.

## 담당 역할

- Next.js 기반 프론트엔드 프로젝트 초기 구조 설계
- 화주, 대리점, 배송기사, 관리자 역할별 화면 흐름 구성
- 백엔드 API 스펙 기반 프론트 API 클라이언트와 타입 설계
- 로그인, 회원가입, 세션 만료 처리, 계정 복구 UI 구현
- 화주 배송 품목 등록/조회/수정 화면 구현
- 대리점 일감 조회와 화주 품목 공개 조회 연동
- 알림 드롭다운, 읽음 처리, unread badge 연동
- 주소 검색, 금액/무게/연락처/사업자등록번호 입력 UX 개선
- API 오류 코드 기반 사용자 안내 UI 구성

## 주요 구현 기능

### 1. 역할 기반 서비스 화면

화주, 대리점, 배송기사, 관리자 역할별로 메뉴와 홈 화면을 분리했습니다.

- 화주: 배송 품목 등록/조회, 계약 요청, 계약 관리
- 대리점: 대리점 프로필, 일감 조회, 제안, 계약 관리
- 배송기사: 기사 프로필, 기사 계약 관리
- 관리자: 전체 기능 접근을 고려한 메뉴 구조

상단 네비게이션과 좌측 사이드바를 공통 `AppShell`로 구성해 권한별 메뉴를 재사용할 수 있게 했습니다. 데스크톱에서는 좌측 메뉴, 모바일에서는 하단 메뉴를 사용하는 반응형 구조를 적용했습니다.

### 2. 인증과 세션 처리

백엔드가 `HttpOnly Cookie`로 access token과 refresh token을 관리하는 구조에 맞춰 프론트에서는 토큰을 직접 읽지 않도록 설계했습니다.

구현 내용:

- 로그인 API는 `credentials: "include"`로 쿠키 저장
- 회원가입과 계정 복구 API는 public API로 `credentials: "omit"` 사용
- 인증 만료 시 refresh 요청 후 원 요청 재시도
- refresh 실패 또는 401 지속 시 메인 화면으로 리다이렉트
- 로그인 후 역할별 프로필 존재 여부 확인
- 프로필 미등록 사용자는 해당 프로필 등록 페이지로 강제 이동

이 흐름을 통해 사용자가 권한은 있지만 필수 사업 정보를 등록하지 않은 상태에서 주요 기능에 접근하는 문제를 줄였습니다.

### 3. 화주 배송 품목 관리

화주가 계약 요청에 사용할 배송 품목을 등록하고 조회할 수 있는 화면을 구현했습니다.

구현 필드:

- 품목 카테고리
- 품목명
- 설명
- 평균 상품 가격
- 평균 무게
- 박스 규격
- 배송 목적지 주소
- 파손 위험 여부
- 액체 여부
- 신선식품 여부
- 냉장/냉동 필요 여부

UX 개선:

- 금액 입력 시 `25,000`처럼 천 단위 콤마 표시
- 무게 입력 시 `5,000`처럼 천 단위 콤마 표시
- 박스 규격을 enum select로 관리
- 냉장/냉동 조건을 boolean이 아닌 `NONE`, `REFRIGERATED`, `FROZEN` enum으로 관리
- Kakao 우편번호 검색을 통한 주소 입력
- 프로필 미등록 시 빨간 오류 대신 프로필 등록 CTA 안내

### 4. 대리점 일감 조회

대리점이 화주가 공개한 계약 요청을 조회할 수 있는 일감 조회 화면을 구현했습니다.

표시 정보:

- 상품명
- 상품 카테고리
- 월 물량
- 희망 단가
- 박스 크기
- 집하 시간
- 토요일 배송 필요 여부
- 반품 필요 여부
- 픽업 주소
- 메모

추가로 대리점이 특정 화주의 등록 배송 품목을 확인할 수 있도록 공개 품목 조회 API를 연동했습니다.

- `GET /api/v1/vendors/{vendorId}/products`
- 대리점 권한으로 화주 품목 목록 조회
- 평균가, 무게, 박스 규격, 온도 조건, 목적지 주소 표시
- 추후 대리점 직접 계약 요청에서 품목 선택 흐름으로 확장 가능

### 5. 알림 기능

상단 네비게이션에 알림 버튼을 추가하고 백엔드 알림 API와 연동했습니다.

구현 내용:

- unread count badge 표시
- 60초 단위 polling
- 알림 드롭다운 목록 조회
- 개별 알림 클릭 시 읽음 처리
- `linkUrl`이 있으면 해당 화면으로 이동
- 전체 읽음 처리
- 실패 시 API error message 표시

실시간 SSE/WebSocket이 없는 현재 백엔드 조건에서 REST polling 방식으로 구현했습니다.

### 6. 계정 복구

아이디 찾기와 비밀번호 재설정 기능을 별도 계정 복구 화면으로 구현했습니다.

구현 탭:

- 아이디 찾기
- 비밀번호 인증 메일 요청
- 인증 토큰으로 비밀번호 변경

보안/UX 고려:

- 계정 존재 여부를 프론트에서 구분하지 않음
- 백엔드의 동일 응답 메시지 정책을 그대로 반영
- 비밀번호 재설정 토큰 TTL 1분 안내
- 새 비밀번호와 확인 값 불일치 시 말풍선 형태 오류 표시
- 성공 후 자동 로그인 없이 다시 로그인하도록 안내

### 7. 입력 UX 개선

서비스형 폼에서 자주 발생하는 입력 오류를 줄이기 위해 프론트 단에서 포맷팅과 검증을 추가했습니다.

- 사업자등록번호: `123-45-67890` 형식 자동 포맷, 10자리 제한
- 연락처: `010-1234-5678` 형식 자동 포맷, 10~11자리 검증
- 금액: 화면 표시용 천 단위 콤마
- 무게: 화면 표시용 천 단위 콤마
- 주소: Kakao 우편번호 검색으로 우편번호/주소/지역 자동 입력
- 비밀번호 확인: 불일치 시 submit 방지와 말풍선 오류 표시

## 기술적 의사결정

### API 클라이언트 공통화

모든 API 응답이 `payload.code`, `payload.errorMessage`, `payload.response` 구조로 내려오는 백엔드 규칙에 맞춰 공통 `apiFetch`를 구현했습니다.

효과:

- API 성공/실패 처리 방식 통일
- 401 refresh retry 로직 중앙화
- API별 컴포넌트에서 중복 fetch 코드 감소
- 백엔드 error code 기반 사용자 안내 가능

### 타입 중심 API 연동

백엔드 스펙이 자주 변경되는 상황에서 enum과 request/response 타입을 명시해 프론트 변경 지점을 빠르게 찾을 수 있도록 했습니다.

예시:

- `ProductCategory`
- `BoxSize`
- `ColdChainType`
- `Carrier`
- `ContractRequestType`
- `ContractRequestStatus`

### 프로필 선등록 흐름

화주/대리점은 계정만 있다고 바로 주요 기능을 사용할 수 없고, 사업자 정보나 영업 거점 정보가 필요합니다. 로그인 직후 프로필 조회 API를 호출해 프로필이 없으면 등록 화면으로 이동하도록 구현했습니다.

이를 통해 다음 문제를 방지했습니다.

- 배송 품목 등록 시 `VENDOR_NOT_FOUND` 발생
- 대리점 일감 조회 시 `AGENCY_NOT_FOUND` 발생
- 사용자가 어떤 정보를 먼저 등록해야 하는지 모르는 문제

### feature flag friendly UI

대리점의 화주 품목 공개 조회 API는 백엔드 정책 토글로 비활성화될 수 있습니다. 프론트는 `AGENCY_PRODUCT_PUBLIC_READ_DISABLED` 같은 서버 오류 메시지를 그대로 표시하도록 해 정책 변경에 대응할 수 있게 했습니다.

## 트러블슈팅과 개선 사례

### 1. HttpOnly Cookie 인증 처리

초기에는 토큰을 JS에서 읽어야 하는지 혼동이 있었지만, 백엔드 인증 방식이 HttpOnly Cookie임을 기준으로 정리했습니다.

해결:

- 프론트에서 access token, refresh token을 읽지 않음
- 모든 보호 API에 `credentials: "include"` 적용
- 회원가입/계정 복구 같은 public API는 `credentials: "omit"` 적용

### 2. 프로필 미등록 404 문제

회원가입 후 배송 품목 등록 시 화주 프로필이 없어 `VENDOR_NOT_FOUND`가 발생했습니다.

해결:

- 회원가입과 프로필 등록을 분리
- 로그인 후 프로필 존재 여부 확인
- 프로필이 없으면 등록 페이지로 이동
- 각 기능 화면에도 프로필 등록 안내 CTA 추가

### 3. 변경되는 API 스펙 대응

백엔드에서 `requiresColdChain: boolean`이 `coldChainType: enum`으로 변경되고, `boxSize`도 문자열에서 enum으로 바뀌었습니다.

해결:

- API 타입을 명시해 변경 영향 범위 확인
- 폼 입력을 select 기반 enum으로 변경
- 화면 label과 서버 enum 값을 분리

## 포트폴리오에서 강조하면 좋은 점

### 서비스 이해도

단순 CRUD 화면이 아니라 택배 계약 시장의 문제를 이해하고, 화주와 대리점이라는 서로 다른 사용자 역할의 흐름을 분리해 설계했습니다.

### API 스펙 기반 개발 역량

백엔드 API가 계속 변경되는 상황에서 handoff 문서를 기반으로 request/response 타입, 인증 방식, 오류 코드를 빠르게 반영했습니다.

### 인증/권한 UX

HttpOnly Cookie 인증, refresh retry, 세션 만료 리다이렉트, 역할별 프로필 강제 등록 같은 실제 서비스에 필요한 인증 흐름을 구현했습니다.

### 사용자 입력 품질 개선

사업자등록번호, 연락처, 금액, 무게, 주소 입력처럼 실제 운영 서비스에서 중요한 폼 UX를 세밀하게 다뤘습니다.

### 확장 가능한 화면 구조

공통 AppShell, 역할별 메뉴, feature별 API 파일, typed response 구조를 통해 기능이 늘어나도 유지보수하기 쉬운 구조를 만들었습니다.

## 이력서용 요약 문장

택배 계약 플랫폼 `택배왕`의 프론트엔드를 Next.js와 TypeScript로 구현했습니다. 화주, 대리점, 배송기사 역할별 화면 구조를 설계하고, HttpOnly Cookie 기반 인증 흐름, 프로필 선등록, 배송 품목 관리, 대리점 일감 조회, 알림, 계정 복구 기능을 백엔드 API와 연동했습니다. 반복 입력이 많은 물류 서비스 특성을 고려해 주소 검색, 금액/무게 포맷팅, 사업자등록번호/연락처 자동 포맷팅 등 폼 UX를 개선했습니다.

## 면접 답변용 핵심 포인트

### Q. 이 프로젝트에서 가장 신경 쓴 부분은?

권한과 상태에 따른 사용자 흐름입니다. 화주와 대리점은 로그인만으로 모든 기능을 쓸 수 없고, 각각 프로필이 있어야 배송 품목이나 일감 조회를 사용할 수 있습니다. 그래서 로그인 후 프로필 존재 여부를 확인하고, 없으면 등록 페이지로 이동하도록 구현했습니다. 기능 화면에서도 프로필 미등록 상태를 단순 오류가 아니라 CTA가 있는 안내로 보여주도록 개선했습니다.

### Q. 기술적으로 어려웠던 부분은?

백엔드 API 스펙이 계속 변경되는 상황에서 프론트 타입과 UI를 일관되게 유지하는 부분이 어려웠습니다. 예를 들어 냉장/냉동 조건이 boolean에서 enum으로 바뀌고, 박스 크기도 free-form string에서 enum으로 변경됐습니다. 이를 타입으로 명확히 정의하고, 화면 입력도 select 기반으로 바꿔 서버와 프론트의 데이터 불일치를 줄였습니다.

### Q. 실제 서비스 관점에서 고려한 점은?

HttpOnly Cookie 인증 구조를 유지해 토큰을 JS에서 직접 다루지 않도록 했고, 401 발생 시 refresh 후 재시도하는 공통 API 클라이언트를 만들었습니다. 또한 세션이 만료되면 메인 화면으로 리다이렉트하도록 해 사용자가 권한 없는 화면에 머무르지 않게 했습니다.

### Q. 앞으로 개선하고 싶은 점은?

- 대리점 제안 제출 화면 구현
- 화주가 제안을 비교하고 수락하는 화면 구현
- 계약 상세/상태 변경 흐름 구현
